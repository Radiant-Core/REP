## Abstract

This document provides a comprehensive overview of the Radiant blockchain architecture, including its unique features, design principles, and technical components. It serves as a reference for developers, researchers, and contributors to understand the system's design decisions and architectural patterns.

## Overview

Radiant is a high-performance blockchain designed for digital asset creation and transfer. It builds upon proven blockchain technology while introducing innovative features for scalability and expressiveness.

### Key Architectural Principles

1. **Performance First**: Optimized for high throughput and low latency
2. **Expressive Programming**: Rich scripting capabilities for complex applications
3. **Backward Compatibility**: Maintains compatibility with existing Bitcoin patterns
4. **Decentralization**: Minimizes centralization vectors and promotes network health
5. **Security**: Prioritizes cryptographic security and attack resistance

## Core Components

### 1. Consensus Layer

#### Proof of Work
- **Algorithm**: SHA512/256
- **Block Time**: 5 minutes
- **Difficulty Adjustment**: ASERT (Asymptotically Scheduled Epoch Reduction)
- **Target Block Size**: 256 MB

#### Transaction Processing
- **Version 3 Transactions**: Enhanced transaction format with induction proofs
- **Unique Reference Model**: Digital asset authenticity via OP_PUSHINPUTREF
- **Signature Hashing**: Output-hash segmentation for improved security

### 2. P2P Network Layer

#### Protocol Features
- **Message Size Limit**: 8 MB protocol messages
- **Block Propagation**: Optimized for large blocks
- **Peer Management**: Advanced peer selection and scoring
- **Future**: BIP324 encryption support (REP-0004)

#### Network Discovery
- **DNS Seeders**: Reliable peer discovery (REP-0001)
- **Peer Exchange**: Efficient peer sharing
- **Connection Management**: Adaptive connection strategies

### 3. Memory Pool

#### Mempool Policy
- **Transaction Validation**: Comprehensive validation rules
- **Fee Estimation**: Dynamic fee calculation
- **Priority Ordering**: Multi-dimensional transaction prioritization
- **Size Limits**: Configurable mempool size constraints

#### Policy Features
- **RBF (Replace-By-Fee)**: Fee bumping support
- **CPFP (Child-Pays-For-Parent)**: Child transaction fee sponsorship
- **Anchoring**: Transaction anchoring for fee estimation

### 4. Storage Layer

#### Database Architecture
- **LevelDB**: Primary storage engine
- **Future**: RocksDB migration consideration (from roadmap)
- **UTXO Set**: Efficient UTXO management
- **Block Storage**: Optional full historical block storage

#### Node Profiles
- **Archive**: Full history and indexing
- **Agent**: Pruned, UTXO-focused
- **Mining**: Recent blocks, optimized for mining

### 5. RPC Interface

#### JSON-RPC API
- **Standard Bitcoin RPC**: Compatible with existing tools
- **Radiant-Specific RPC**: Enhanced functionality
- **Async Support**: Future async RPC server (REP-0003)
- **Authentication**: Secure RPC access control

#### Key RPC Categories
- **Blockchain**: Block and transaction queries
- **Mining**: Mining and block template RPCs
- **Network**: Peer and network status
- **Wallet**: Key management and transactions
- **PSRT**: Atomic swap protocol operations

## Unique Radiant Features

### 1. PSRT (Glyph Swap Protocol)

#### Overview
PSRT enables on-chain atomic swaps without requiring trusted intermediaries. It's implemented via the `-swapindex` flag and provides:

- **Trustless Swaps**: No counterparty risk
- **On-Chain Order Book**: Transparent order discovery
- **Price Discovery**: Market-based price formation
- **Cross-Asset Support**: Multiple asset types

#### Technical Implementation
```cpp
// PSRT order structure
struct PSRTOrder {
    uint256 assetId;
    uint256 wantAssetId;
    uint64_t amount;
    uint64_t wantAmount;
    uint256 makerPubkey;
    int32_t blockHeight;
    uint256 signature;
};
```

### 2. Unique Reference Model

#### OP_PUSHINPUTREF
Radiant introduces OP_PUSHINPUTREF for digital asset authenticity:

- **Asset Creation**: Bind digital assets to unique references
- **Ownership Proofs**: Cryptographic proof of asset authenticity
- **Transfer Tracking**: Maintain asset provenance
- **Anti-Counterfeiting**: Prevent duplicate asset creation

### 3. Enhanced Scripting

#### Script Capabilities
- **Bitcoin Compatibility**: All Bitcoin opcodes supported
- **Radiant Extensions**: Additional opcodes for advanced features
- **Future Covenants**: CTV support planned (REP-0005)
- **Performance**: Optimized script execution

## Node Architecture

### Node Types

#### Full Node (Archive)
- Complete blockchain history
- Full transaction indexing
- RPC server capabilities
- PSRT order book maintenance

#### Pruned Node (Agent)
- UTXO set only
- Recent blocks (~10,000)
- Reduced storage requirements
- RPC capabilities

#### Mining Node
- Block template generation
- Mining coordination
- Recent block storage
- Mining-specific optimizations

- **Hash Functions**: SHA256, SHA512/256, RIPEMD160
- **Public Key Crypto**: ECDSA (secp256k1)

### Security Features

- **Script Validation**: Comprehensive script interpreter
- **Consensus Rules**: Strict validation logic
- **Network Security**: DDoS resistance, peer scoring

### Attack Resistance

## Comparison with Other Blockchains

### Bitcoin Similarities
- **UTXO Model**: Unspent transaction output model
- **Proof of Work**: SHA-based mining
- **Script Language**: Bitcoin-compatible scripting
- **P2P Protocol**: Similar network protocol

### Radiant Innovations
- **Larger Blocks**: 256 MB vs 1 MB (original Bitcoin)
- **PSRT Protocol**: On-chain atomic swaps
- **Unique References**: Digital asset authenticity
- **Enhanced Performance**: Optimized for high throughput

### Design Trade-offs

| Feature | Bitcoin | Radiant | Trade-off |
|---------|---------|---------|-----------|
| Block Size | 1-4 MB | 256 MB | Storage vs throughput |
| Script Complexity | Limited | Enhanced | Security vs expressiveness |
| Asset Support | Native | Enhanced | Simplicity vs features |
| Consensus | Conservative | Progressive | Stability vs innovation |

## Conclusion

Radiant's architecture balances innovation with proven technology, providing a robust foundation for digital asset applications while maintaining compatibility with existing blockchain patterns. The modular design allows for incremental improvement through the REP process, ensuring the platform can evolve to meet future needs.

The architecture emphasizes performance, security, and expressiveness, making Radiant suitable for both simple transactions and complex smart contract applications. Ongoing development through the REP process ensures the platform continues to improve while maintaining its core principles.
