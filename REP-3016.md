## Abstract

This REP defines **Induction Proofs**, a cryptographic mechanism for verifying code continuity across transaction chains. Induction proofs enable smart contracts to prove that a series of transactions (P(0), P(1), ..., P(k)) all execute the same code script, establishing trustless verification of contract lineage without requiring full chain traversal at validation time.

## Motivation

### Problem: Code Continuity Verification

When a smart contract creates a chain of transactions (e.g., NFT minting, stateful games, voting), verifiers need to confirm:
1. Each transaction in the chain executed the same code
2. No malicious actor inserted different code mid-chain
3. The chain is valid from genesis to tip

Without induction proofs, this requires:
- Traversing the entire transaction chain
- Storing and comparing all intermediate code scripts
- Expensive verification that grows with chain length

### Goals

1. **O(1) verification time**: Verify chain validity without full traversal
2. **Minimal on-chain data**: Use transaction hash (txid) as proxy for code
3. **Trustless**: No reliance on external indexers or oracles
4. **Composable**: Works with any RadiantScript contract
5. **Deterministic**: Same proof works across all implementations

## Specification

### 1. Core Concepts

#### 1.1 Code Continuity

Two transactions have **code continuity** if:
- They execute identical `codeScript` bytecode
- The codeScript is the program portion of the locking script

#### 1.2 Induction Hypothesis

Given:
- **Base case**: Genesis transaction (n=0) has known-valid code
- **Inductive step**: If transaction k has valid code, and transaction k+1 proves it executed the same code, then transaction k+1 also has valid code

**Proof structure**: P(k) → P(k+1) via txid preimage verification

### 2. OP_PUSH_TX_STATE (0xed)

#### 2.1 Opcode Definition

```
OP_PUSH_TX_STATE (0xed)
Input:  field_number (0, 1, or 2) from top of stack
Output: field value pushed to stack
```

#### 2.2 Field Numbers

| Field | Number | Type | Description |
|-------|--------|------|-------------|
| `tx.state.txId` | 0 | bytes(32) | Transaction hash (txid) |
| `tx.state.inputSum` | 1 | int | Sum of all input values |
| `tx.state.outputSum` | 2 | int | Sum of all output values |

#### 2.3 tx.state.txId Behavior

**When nVersion == 3**: The txid is computed as:
```
txid = hash256(CTransactionHashPreimage)
```

Where `CTransactionHashPreimage` is a 112-byte structure containing:
- nVersion (4 bytes)
- hashPrevouts (32 bytes)
- hashSequence (32 bytes)
- hashOutputs (32 bytes)
- nLockTime (4 bytes)
- nExpiryHeight (4 bytes)
- nInputIndex (4 bytes)

This differs from nVersion < 3 where txid = hash256(raw_tx).

### 3. Induction Proof Methods

#### 3.1 Method 1: Reference-Based Induction

Uses `OP_PUSHINPUTREF` (0xd0) and `OP_REQUIREINPUTREF` (0xd8) to chain contracts.

**Mechanism**:
1. Each transaction outputs a UTXO with a unique reference
2. The next transaction requires that reference as input
3. The reference acts as proof of lineage

**Limitations**:
- Requires reference system support
- Cannot verify code identity, only linkage

#### 3.2 Method 2: TxId Preimage Verification (Recommended)

Uses `tx.state.txId` and `OP_PUSH_TX_STATE` to verify code continuity.

**Mechanism**:
1. Input provides previous txid as preimage data
2. Contract extracts `codeScript` from preimage
3. Contract verifies codeScript matches expected hash
4. Contract verifies preimage's txid matches claimed previous transaction

**Code Pattern**:
```cashscript
// Induction proof contract example
contract InductionNFT(
    bytes expectedCodeHash,  // hash256 of valid codeScript
    bytes32 prevTxId
) {
    function mint(sig s, pubkey pk) {
        // Get current transaction's txid
        bytes32 currentTxId = tx.state.txId;
        
        // Verify previous txid matches expected (passed as preimage)
        require(hash256(prevTxId) == expectedCodeHash);
        
        // Verify code continuity via codeScript hash
        require(tx.inputs[0].codeScript.hash256() == expectedCodeHash);
        
        // If we reach here, code continuity is proven
        // Continue with contract logic...
        require(checkSig(s, pk));
    }
}
```

### 5. InductionNFT Example Contract

```cashscript
pragma cashscript ^0.9.0;

/*
 * InductionNFT - NFT with code continuity proof
 * Each mint proves it extends from a valid previous NFT
 */
contract InductionNFT(
    bytes32 collectionRef,    // Collection singleton reference
    int maxSupply,            // Maximum tokens in collection
    bytes expectedCodeHash    // hash256 of this contract's codeScript
) {
    // Mint a new NFT with induction proof
    function mint(
        bytes prevTxId,       // Previous NFT txid (preimage)
        bytes prevProof,      // Merkle proof for prev tx
        sig s,
        pubkey pk
    ) {
        // 1. Verify we're spending a valid input with correct code
        require(tx.inputs[0].codeScript.hash256() == expectedCodeHash);
        
        // 2. Verify prevTxId matches the input's transaction
        require(tx.inputs[0].outpoint.txid == prevTxId);
        
        // 3. Verify we are not exceeding supply
        require(tx.state.inputSum < maxSupply);
        
        // 4. Verify signature
        require(checkSig(s, pk));
        
        // 5. Output new NFT with collection reference
        // (output construction handled by wallet)
    }
    
    // Transfer with continuity preservation
    function transfer(sig s, pubkey pk) {
        require(checkSig(s, pk));
    }
}
```

### 6. Verification Algorithm

#### 6.1 Verifier Steps

Given a chain of transactions T₀, T₁, ..., Tₙ:

```
function verifyInductionChain(chain):
    // Base case: verify genesis
    if !verifyGenesis(T₀):
        return false
    
    // Inductive step: verify continuity
    for i from 1 to n:
        // Get codeScript from Tᵢ's input (which spends Tᵢ₋₁)
        inputCodeScript = Tᵢ.inputs[0].codeScript
        
        // Compute hash
        codeHash = hash256(inputCodeScript)
        
        // Verify against expected (from contract params)
        expectedHash = Tᵢ.contractParams.expectedCodeHash
        if codeHash != expectedHash:
            return false
        
        // Verify txid linkage
        prevTxId = Tᵢ.txData.prevTxId  // from tx.state mechanism
        if prevTxId != Tᵢ₋₁.txid:
            return false
    
    return true
```

#### 6.2 Complexity

- **Time**: O(n) for full chain verification, O(1) for single step
- **Space**: O(1) additional storage per transaction
- **No trusted setup**: All verification is on-chain

### 7. Security Considerations

#### 7.1 Preimage Attacks

**Risk**: Attacker provides fake preimage claiming to be from valid chain.

**Mitigation**:
- `tx.state.txId` is computed deterministically from transaction data
- Input outpoint txid links to actual previous transaction
- Both must match for valid proof

#### 7.2 Code Injection

**Risk**: Malicious input with different codeScript.

**Mitigation**:
- `expectedCodeHash` is contract parameter (immutable at deployment)
- `codeScript.hash256()` is verified against this hash
- Any deviation causes immediate rejection

#### 7.3 Replay Attacks

**Risk**: Valid proof reused in different context.

**Mitigation**:
- Each induction proof is bound to specific transaction inputs
- `tx.state` values are unique per transaction
- `prevTxId` linking prevents cross-chain replay

#### 7.4 Forced Chain Splits

**Risk**: Attacker creates parallel chains with divergent code.

**Mitigation**:
- First valid spend of a UTXO wins (consensus rule)
- Only one chain can be valid per UTXO
- Double-spend protection applies to induction proofs

### 8. Backwards Compatibility

- **v1 Contracts**: No impact, induction proofs are opt-in
- **v2 Contracts**: Can add induction proof fields without breaking existing logic
- **Indexers**: May track `tx.state` usage for analytics, not required for validation
- **Wallets**: Need to support nVersion=3 txid calculation for signing

### 10. Comparison with Alternative Methods

| Method | Verification Cost | Chain Storage | Trust Required | Code Identity |
|--------|------------------|---------------|----------------|---------------|
| **Full Chain Traversal** | O(n) | O(n) | None | Yes |
| **Reference-Based** | O(1) | O(1) | None | No |
| **Induction Proof (this REP)** | O(1) per step | O(1) | None | Yes |
| **Trusted Oracle** | O(1) | O(1) | Oracle | Yes |
| **Zero-Knowledge Proof** | O(1) | O(1) | Setup | Yes |

**Induction proofs provide the best balance**: constant verification time, minimal storage, no trusted parties, and code identity verification.

## Rationale

### Why TxId Instead of Direct Code Hash?

1. **Efficiency**: txid is 32 bytes, codeScript can be arbitrarily large
2. **Uniqueness**: txid includes all transaction data, preventing collisions
3. **Composability**: txid links enable chaining with other proof systems

### Why nVersion=3 Special Handling?

Radiant introduced nVersion=3 for transaction hash preimage separation:
- `hash256(raw_tx)` for transaction relay
- `hash256(CTransactionHashPreimage)` for txid

This allows tx.state.txId to be deterministic while supporting tx malleability fixes.

### Why Not Just Use References?

References (`OP_PUSHINPUTREF`) prove linkage but not code identity:
- Contract A could create a reference, then Contract B could use it
- Induction proofs verify that the SAME code executed at each step

### Why Add to CashScript Instead of Bytecode Only?

High-level language support:
- Type safety: compiler verifies `tx.state.txId` returns bytes(32)
- Readability: `tx.state.inputSum` vs `OP_1 OP_PUSH_TX_STATE`
- Optimization: compiler can optimize multi-field access patterns
